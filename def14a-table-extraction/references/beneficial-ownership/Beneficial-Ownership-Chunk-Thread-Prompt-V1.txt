Process `[CHUNK_FILENAME]` using the bundled V1 plan in `references/beneficial-ownership/Beneficial-Ownership-Agent-Plan-V1.md`.

Execution rules:
- Read the plan first and use it as the source of truth for extraction, schema mapping, local HTML lookup, footnote capture, cumulative footnote allocation, directness, confidence, and report behavior.
- Read the chunk CSV. Each row is one filing work item, not one beneficial-owner row.
- Use local HTML in `[PROJECT_ROOT]/data/raw` first. If `local_html_path` is present and `local_html_exists=TRUE`, open that file first. Otherwise derive local filenames from CIK and accession number as described in the plan.
- If the filing is not found locally and network access is allowed, download it from `source_filing_url`, save it to `[PROJECT_ROOT]/data/raw/[zero-padded CIK]_[accession-without-dashes].html`, and extract from that saved file. Use a compliant SEC `User-Agent` header for SEC EDGAR downloads, do not overwrite an existing local filing with different content unless explicitly instructed, and mark the filing `unresolved` if the download fails.
- Treat each filing as independent. Do not reuse table or footnote mapping assumptions across filings, even for the same issuer.
- Locate the real beneficial ownership body table. Ignore table-of-contents links, Section 16(a) reporting compliance sections, stock ownership guideline sections, equity compensation plan tables, meeting Q&A text, and footnote-only false positives.
- Create row-level output from scratch from the target table.
- Capture row footnote markers and matching footnote content.
- If a table-heading footnote exists, assign it to the first executive/director row, or first owner row if no executive/director row exists.
- If one footnote describes multiple named people, allocate amounts person by person only to matching rows.
- Analyze `Footnote Content` to populate direct shares, options exercisable within 60 days, RSUs vesting within 60 days, warrants exercisable within 60 days, other included securities, `Is Direct`, and `Includes Indirect Ownership`.
- Set `Includes Indirect Ownership=YES` whenever the footnote indicates the table amount includes any non-direct ownership component, including options, RSUs, warrants, restricted stock, stock awards, convertible/issuable securities, trusts, family members, entities, partnerships, funds, or similar arrangements. Use `NO` only when no indirect or included ownership component is indicated.
- Keep the table total in `Shares Beneficially Owned`; do not overwrite it with parsed component values.
- Extract every confidently disclosed numeric included-security component into the most specific numeric component column. Put miscellaneous included security amounts that do not fit any dedicated component column in `Other Included Non-Direct Securities Amount` and explain them in `Other Included Securities`.
- If multiple miscellaneous included security amounts do not fit the dedicated columns, sum only those miscellaneous included amounts into `Other Included Non-Direct Securities Amount`. Do not include amounts already mapped to options, RSUs, warrants, restricted stock/stock awards, convertible/preferred shares, indirect common shares, or excluded/disclaimed securities.
- Put securities that are mentioned but excluded, disclaimed, capped out, blocker-limited, or otherwise not included in the table total into `Excluded / Disclaimed Securities Amount` and `Excluded / Disclaimed Securities Detail`, not into included component columns.
- Leave `Total Included Non-Direct Amount` and `Calculated Direct Shares` blank. Those calculation fields are populated later after merge; do not use them for worker-side arithmetic.
- Use audit-level precision. If uncertain, do not guess; set `investigation_required=TRUE` in the report and explain the issue in `error`.
- Do not create helper scripts or code files in the workspace unless explicitly asked. If temporary code is necessary, use inline commands or `/tmp`.
